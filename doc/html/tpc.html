1Introduction

The purpose of this document is to list the places where the OSDL Database Test 2 (OSDL-DBT-2) test kit deviates from the TPC-C specification.

2Database Design

The kit currently does not enforce any integrity constraints whatsoever on the database.

3Database Scaling

By default the datagenerator for the database will scale the database properly based on the number of warehouses built. The cardinality for all the tables, except DISTRICT and ORDER_LINE, can be manually overridden.

4.2Payment

In the case there customer information is selected by c_last, there can be more than one row of customer information returned. When there is more than one row of customer information, the customer at n/2 is selected, where n is the total number of rows returned. The stored procedure currently selects the first row returned. Also, the customer is supposed to be seelcted by c_w_id and c_d_id. The stored procedure curerntly does not use c_w_id and c_d_id.



In the case where the selected customer has bad credit, the original c_data is not appending to the new c_data updated for the customer.

4.3Order-Status

The input data generator does not generate data for the case where a customer is select by c_last. It only generates data to select a customer by c_id.

4.4Delivery

Statustics are not collected for the case when deliveries are not made for a district.



Delivery transactions are not queued but are executed immediately.

5Performance Metrics

The perforamnce of the kit is measured by the number of New-Order transactions execute per second.



The mix of the transactions is complient to the TPC-C sepcification by default, but it can be adjusted outside the restrictions placed in the specification. (I don't know why this is explained under the respective section in the TPC-C specification but I plan to reorganize this entire document.)

6Driver

The driver is currently designed to work with a client program to reduce the number of connections to the database. The driver can be used in a way such that it implements most of the client code directly to reduce the number of connections to the database from the driver program directly, instead of openning a database connection per terminal emulated.
